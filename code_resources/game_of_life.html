From Traffic Simulations to Conway's Game of Life: Exploring Agent-Based Modelling with Rust and WebAssembly

When I started a role working with agent-based modelling (ABM) to simulate traffic networks using MATSim, I found myself wanting to understand more about the modelling philosophy behind these systems. Large-scale simulations can feel complex and opaque—thousands (or millions) of agents making local decisions that produce global traffic patterns, congestion waves, and emergent behaviour.

To get a better feel for where many of these ideas originated, I went back to a classic example of emergent systems: Conway's Game of Life. As a learning exercise—and partly out of curiosity about performance and WebAssembly—I built it in Rust and compiled it to run in the browser via WASM.

This post covers that small project: why I chose Game of Life, what I learned building it, and how it sharpened my intuition for agent-based models.

Cellular Automata, Local Rules, and Emergent Behaviour

At its core, the Game of Life is a cellular automaton: a grid of cells that evolve over time according to a simple set of deterministic rules. Each cell is either alive or dead, and its next state depends only on its immediate neighbours. There is no central controller or global objective—only local interactions.

And yet, richer structures emerge:

  • oscillators
  • gliders
  • stable formations
  • chaotic patterns that somehow self-organise

This is what drew me in. Even though it is a toy model, it highlights a core concept I also see in traffic simulations: simple local rules can produce complex and often surprising global behaviour.

In MATSim, agents make local travel choices—departure time, route selection, mode choice—based on their own objectives and constraints. From that, we see network-level effects: congestion, bottlenecks, feedback loops, and system equilibria. The difference is scale and realism, but conceptually both systems share a lineage. Game of Life offers a compact, intuitive way to explore those ideas on a grid small enough to reason about.

Why Rust + WebAssembly?

I also wanted this to be an opportunity to learn something new technically.

  • Rust appealed because of: strong guarantees around memory safety, explicit state management, and good performance for simulation-style workloads.
  • WebAssembly made it possible to: run the simulation in the browser, get native-like performance without JavaScript overhead, and package the project as an interactive demo rather than a static write-up.

A browser-based simulation feels appropriate for this kind of model—you can tweak grid sizes, step speeds, or initial states and immediately see the dynamics unfold.

Rust + WASM also forced me to think carefully about how state is represented, how data moves across the Rust ↔ JavaScript boundary, and where performance bottlenecks actually occur. Even on something as small as a Game of Life grid, those questions matter.

Designing the Simulation

Representing the grid

I experimented with a 2D array and a flattened 1D vector. I eventually settled on a flat vector for performance and simplicity; contiguous buffers map well to the WASM memory model.

Updating cell state

For each tick I:

  • compute the number of live neighbours
  • apply the classic Game of Life rules
  • write results into the next buffer

Instead of mutating the grid in place, I alternate between two buffers each step. That makes the logic easier to reason about and avoids subtle ordering bugs where an updated cell influences neighbouring updates in the same tick.

Performance considerations

Even in a small experiment like this, I noticed that allocations inside the update loop add up, neighbour-counting dominates compute time, and naive bounds checks create overhead. Rust makes those tradeoffs explicit—useful practice for larger modelling systems.

Rust → WebAssembly → JavaScript

The most interesting part was the interface. Questions I had to think through included:

  • What state should live in Rust vs in JS?
  • Should the browser render from raw memory, or from a copied buffer?
  • How often do I cross the WASM boundary per animation frame?

This was great practice in thinking about data ownership, memory layout, and the cost of abstraction—all skills that transfer directly to more complex simulation work.

What This Taught Me About Agent-Based Modelling

Local rules really do scale into global dynamics

Even when the rules feel trivial, the resulting system is hard to predict, develops structure over time, and sometimes converges, collapses, or explodes. This maps surprisingly well to real-world simulations. In MATSim, agents do not coordinate globally; their decision rules are local and bounded, yet outcomes—congestion patterns and travel waves—look systemic.

The model feels simple, but interpreting outcomes is not

Small changes in initial conditions cascade. Intended outcomes are not guaranteed, and structure can emerge accidentally. A model is not just an implementation—it is a lens for interpreting emergent behaviour. That mindset applies to urban and transport simulations as much as to grid-based automata.
